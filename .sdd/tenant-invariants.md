# テナント（組織）不変条件

API の認可・データの所属関係で迷ったときの**正本**。平易に書くが、サーバ・DB それぞれの役割ははっきり分ける。

**組織の壁は常にサーバ（API）が張る。** ブラウザで一覧を会社で絞って表示するだけでは、セキュリティ上のテナント分離にならない。

---

## (A) 非スーパーアドミン

- ログインユーザーがスーパーアドミンでないとき、**どのデータを返していいか・更新していいかは、JWT に入っている `organization_id` だけを見て決める**（これが正）。
- 返していいのは、その組織に属するデータだけ。URL やクエリで他社の ID を指定したら **403 か 404** にする（どちらに揃えるかは [api-spec.md](api-spec.md) の各エンドポイントどおり）。
- サーバが返す JSON から組織条件を抜いて「全部返す」ような実装はしてはいけない。
- フロントが会社 ID でフィルタして見せるのは **UI の都合**であり、**テナント分離の代わりにはならない**（非スーパーアドミンはサーバが JWT で必ず弾く）。

---

## (B) 親子のデータ（例: ステータス → ワークフロー）

### API がやること

- 例: `GET /workflows/:id/statuses` では、**先に** `:id` のワークフローが「今の JWT の組織のものか」を調べる。**違えば 403 か 404 で終わり。** ここで止める。
- それが自分の組織のワークフローだと分かったら、そのあとは **`workflow_id` がその `:id` の行だけ**返せばよい。別途「会社 ID でもう一度絞る」は、設計上の必須ルールではない（親のチェックが通っていて DB が正しければ、結果はその会社にだけ閉じる）。

### DB が正しいこと

- ステータス行の `workflow_id` が、別会社のワークフローを指しているような行は **DB が壊れている状態**とみなす。
- 同一ワークフロー内で `(name, type, order)` が重複する行はデータ欠陥。**起動時マイグレーション**（`MigrateStatusDedupeAndUniqueIndex`）で既存重複を付け替え・削除し、**部分ユニークインデックス**で再発を防ぐ。API は正しい DB から行をそのまま返す。

---

## (C) やってはいけないこと

- テナントを守ったことにするために、**レスポンスの形を整えるだけ**（重複をまとめる、ブラウザ側で隠すだけ）に頼らない（DB 側で整合性を取る）。

---

## (D) スーパーアドミン

- スーパーアドミンは JWT に「自分の会社」がないので、一覧 API は **パラメータを付けないと全組織ぶん返る**ことがある。
- 画面で「いま選んでいる1社だけ」を見せたいときは、**フロントはクエリで `org_id` を付けてリクエストする**。大事なのは **サーバがその `org_id` に従って、返す行をその会社だけに絞る**こと。フロントが全件を取ってから会社で絞るのは、**スーパーアドミン画面のセキュリティの代わりにはならない**（改ざんされたクライアントを想定するなら、絞りはサーバ必須）。
- 全組織ぶん返す呼び方は、「全社を横断する画面」など用途がはっきりしたところだけにする。単一社の管理画面では **`org_id` 付きで API を呼ぶ**。同じ画面のコードから、意図せず `org_id` なしで全件取得しないようにする（仕様とコードの対応は [api-spec.md](api-spec.md) と [architecture.md](architecture.md) に書く）。
