# 設計原則

## 最優先原則：開発はローカル、運用はクラウド

**本プロジェクトの設計・実装において、常に最上位で考慮すべき原則である。**

- **開発環境**: ローカル PC（WSL + Docker + Go + Node.js）
- **運用環境**: GCP（Google Cloud Platform）等のクラウド

**AI はこの前提を常に考えて提案すること。** プランニング・設計・実装のすべてにおいて最上位で考慮する。  
（WSL 移行理由・Cursor 接続必須など、その他の最重要事項は [AGENTS.md](../AGENTS.md#最重要事項人ai-ともに必ず認識すること) を参照）

この原則に基づき、以下を満たす設計・実装を行うこと:

- ローカルで開発・テストが完結すること
- クラウド（GCP）へのデプロイを前提とした構成にすること
- 環境差異を吸収する設計（環境変数、設定の分離）とすること
- 既存の Dockerfile は GCP のコンテナビルド・デプロイにそのまま利用可能であること

---

## プロダクト方針（Issue 管理）

本プロジェクトは **Issue 管理システム**の構築を主目的とする。稟議・承認ワークフロー専用のドメインは設計の中心に置かない。**誰がステータスを変更してよいか**は [transition-permissions.md](transition-permissions.md) で定義する（承認とは別物として扱う）。

## データベースの一意制約（PK のみ）

**PostgreSQL 上の一意制約は各テーブルの主キー（PK）に限定する。** 組織名の一意、組織内メールの一意、プロジェクトキーの一意、同一ワークフロー内の (名前, 表示順) の一意など、業務上の「この組み合わせは1件だけ」は **DB の `UNIQUE` や部分ユニークインデックスでは表現せず、Service 層で検査・拒否する。**

- 検索・JOIN 用に **非一意のインデックス** を張ることは許容する（パフォーマンス目的）。
- 同時更新による競合は、必要に応じてトランザクションや行ロックで抑える（設計は [db-schema.md](db-schema.md)・[layer-responsibility.md](layer-responsibility.md) に沿う）。

詳細と列ごとのルールの正本は [db-schema.md](db-schema.md) を参照する。
