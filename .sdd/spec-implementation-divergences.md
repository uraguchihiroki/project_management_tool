# 仕様と実装の乖離（明示リスト）

`.sdd` を正としたとき、**実装がまだ追いついていない点**だけを列挙する。設計の正本は各ドキュメントに任せる。

ここに載っている項目は **ユーザーが内容を認識し、現時点で許容している乖離** とする。未合意のズレは書かない。新たな乖離を検知したら [AGENTS.md](../AGENTS.md) の **§7** に従い、ユーザーに確認してから追加する。

## 運用ルール（状態列）

表の **`状態`** 列は次のとおり。

| 値 | 意味 |
|----|------|
| `open` | 追跡中（乖離の解消・方針確定をまだ終えていない、または許容として監視中）。 |
| `done` | 解消済み、または許容として確定し **このリストでの追跡を終えた**（履歴として行は残してよい）。 |

- 新規に行を足すときは原則 **`open`**。
- **`done`** に更新した事実は、コミット・PR・チャットなどでいつ誰が変えたか分かればよい（本ファイルに日付列は設けない）。

## スコープ合意（一文）

論理削除を `.sdd/db-schema.md` に沿って全テーブル・全 DELETE 経路へ適用する **対象の切り方・復元の要否・API の意味** は、別途詰めたうえで実装する。本ファイルは **現状の乖離の明示** に用いる。

## 読み方（このリストで言う「乖離」）

- **「テーブルに `deleted_at` が無い」ことだけ**を指しているわけではない。業務モデルには GORM の `DeletedAt` を付け、通常の `Delete` はソフト削除に寄せてある。
- ここで問題にしているのは、`.sdd/db-schema.md` の **「削除は UPDATE 相当（論理削除）」という理想**に対して、実装上 **どうしても別の削除の仕方になる経路**が残っていること、および **起動時の移行処理**が業務 API とは別のルールで `DELETE` を使うこと。

## 合意済みの乖離

| 項目 | 内容 | 状態 |
|------|------|------|
| 論理削除まわりの残差 | **起動時マイグレーションのみ** … サーバ起動時に [`internal/db`](../backend/internal/db) が実行する処理（レガシー列の除去・重複 statuses のマージ・旧業務 UNIQUE の DROP・任意の非一意インデックス作成・結合テーブルの代理 PK 移行・`workflow_transitions` の重複行畳み込みなど）は、**業務 API ではなく DB 移行・整合用**である。ここでは意図的に **生 `DELETE`** や **`Unscoped().Delete`** が使われる（[`cmd/server/main.go`](../backend/cmd/server/main.go) の起動順で呼ばれる）。**業務 Repository**（結合テーブルの付け替え、`workflow_transitions` / `project_status_transitions` の許可遷移の更新含む）は原則 **ソフト削除＋新行 `Create`** であり、**`Unscoped` による物理削除**は業務経路では使わない。業務上の一意は DB の UNIQUE ではなく Service で保証する（[principles.md](principles.md)）。詳細は [db-schema.md](db-schema.md) の論理削除ノート参照。 | open |

## インシデント記録: ワークフロー一括保存（PUT editor）と是正

開発・検証環境で実際に観測された事象と、**是正すべき処置**をセットで記す。**セキュリティに踏み込む対策は別フェーズでよい**（後回し）とするが、運用上の空白はここで明示する。

### 起こったこと（事実）

- `workflow_transitions` の複数行が **同一タイムスタンプ（論理削除の `deleted_at`）**で更新され、アクティブな遷移本数が減ったように見えた。
- 利用者側では **「許可遷移を一覧から一本ずつ削除した」操作はしていない**一方、DB 上は論理削除が付いていた。

### 技術的な説明（バグというより実装の仕様）

- [`WorkflowEditorService.Save`](../backend/internal/service/workflow_editor.go) は、トランザクション内で **当該 `workflow_id` の遷移を一括で論理削除**したうえで、リクエスト本文の `transitions` を **新規行として再作成**する。
- そのため **保存が 1 回成功するたび**、置き換え対象となった旧行には **同じトランザクション時刻の `deleted_at`** が付く。これは **「人間がミリ秒まで揃えて 6 レコードを個別削除した」わけではなく、アプリの一括置換パス**でも起きる。
- 管理画面の **下部ストラップ一覧**は `editorDraft.transitions`（＝取得した API の配列）をそのまま表示する。**一覧の行数が DB の生 COUNT より少ない**ように見える場合は、まず **`GET /api/v1/workflows/:id/transitions` の返却件数**と、**論理削除行を含めた SQL の数え方**（`workflow_id`・`deleted_at`）を揃えて比較する。
- `Save` 内には **開始ステータスから全ステータスへの到達可能性**チェックがある（ルールを満たさないペイロードはコミットされない）。**「明らかに到達不能なのに通った」**と疑う場合は、**当該リクエストの `statuses` / `transitions` の実体**（Network の生 JSON）と突き合わせる。

### 問題となりうる点（設計上のギャップ）

- 認可と **スキーマ・業務ルール**（開始 1 件、到達可能性、重複ペア禁止など）を満たす限り、**クライアントが送ったスナップショットがサーバー上の正**になる。**改ざん・誤送信・古いタブのドラフト**がルール内なら上書きされ得る。これは **ゼロトラスト的な「送付データの検証」ではない**。

### 是正すべき処置（優先度の目安）

| 優先 | 処置 | 備考 |
|------|------|------|
| 高（セキュリティ以外） | **監査ログ** … `PUT .../editor` 成功時に、実行者（JWT `user_id`）、`workflow_id`、組織、**保存前後の遷移件数**（または論理削除に入った件数と再作成件数）などを **サーバー標準出力または運用ログ基盤**に残す。 | ログフォーマット・集約基盤の設計は別タスクでよい（実装フェーズを後ろにずらした場合でも、**要件としてここに残す**）。 |
| 中 | **楽観ロック** … ワークフロー（または editor 用）に **版**（`updated_at` / revision / ETag）を持たせ、**クライアントが持つ版と DB が一致しない PUT は 409**。 | **同時編集・古いタブの誤上書き**対策が主目的。 |
| 後回し（セキュリティ寄り） | 改ざん検知、リクエスト署名、クライアント非信頼モデルへの移行など。 | **ユーザー方針: セキュリティに踏み込む強化は後フェーズ**。 |

**状態:** `open`（上記の「高」「中」を解消または別文書に移すまで本セクションで追跡する）。
