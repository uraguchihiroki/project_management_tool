# ステータス遷移の権限（設計メモ・候補比較）

Issue 管理を主目的とするにあたり、**稟議・承認（誰が許可したかの証跡）とは別**に、ステータス遷移について **運用上のリスク**（例: 想定外の人が完了へ動かした）にどう対処するかを決める。

- **本設計の方針**: **遷移のたびに「誰だけが実行していいか」で API を拒否する**ゲートキーパー型は **採用しない**。代わりに **遷移アラート**（想定外の実行者への通知）と **インプリント**（事後監査）で **確認を促す**。
- **目的の例**: チケットが想定外の経路・想定外の人によって Close されたとき、**関係者が気づき、確認・是正できる**こと。**Close を特定ユーザー ID にだけ結び付けない**こと（人事異動で権限が個人に張り付くと運用が破綻しやすい）。
- **要件の例**: 「**営業部の課長**相当のみが完了遷移の想定アクター」など、**グループ × 役割（名称は未確定）**で **想定と実態のずれ**を表現し、ずれたときに **遷移アラート** が飛ぶようにする。

用語は **「役職」に固定しない**。階層付きグループ・名前付きグループ・グループスコープなど、下記を議論のうえ確定する（**第1案はまだ決めない / 1-D**）。

## 5. データ層（[db-schema.md](db-schema.md)）に含める仕様

README の読書順 **5** に対応。**クエリしやすい**ことを優先する。

| 項目 | 方針 |
|------|------|
| **インプリント** | 操作ごとの **事実の1記録**（`issue_events` の1行）。後から **組織・Issue・種別・期間・操作者** で絞れる。**時刻**は属性として **`occurred_at`（`TIMESTAMPTZ`）**。通常の `created_at` / `updated_at` と役割を分ける。定義は [db-schema.md](db-schema.md) の **インプリント**。 |
| **イベントログ** | **`issue_events`** は **追記のみ**の **インプリント**列。ステータス変更・担当変更等で **誰が・いつ・当時の担当** を残し、監査クエリ（例: 完了時に actor と assignee が同一）に使う。 |
| **グループ** | **`groups`**（組織スコープ）と **`organization_user_groups`** 多対多。 |
| **Position 専用テーブル** | 必須としない。 |

## 7. 遷移・遷移アラート・監査（本ドキュメントの本丸）

読書順 **7**。ここでは **意味**を固定する（表は db-schema / ルートは api-spec）。

### 三層（承認システムではない）

| 層 | 内容 | 例 |
|----|------|-----|
| **① 許可される遷移（形）** | Redmine のように **from→to の列挙**で、未着手→…→完了の **経路を強制**したり、バイパスを禁止したりする。 | 作業中からいきなり完了へ不可、等。 |
| **② 遷移アラート** | 遷移が発生したときに、運用で定義した条件に応じて、**メール等で通知**する。 |
| **③ 監査** | システムで **作業者＝確認者を必ず分けない**運用も許容しつつ、**事後**に **インプリント列**（`issue_events`）で **同一だった遷移**を検索する（[db-schema.md](db-schema.md) のスナップショット列と対）。 |

### 遷移アラートとは

- **定義**: ステータス遷移が行われたときに、運用で定義した条件に応じて、**メール等で関係者へ通知**する仕組み。
- **目的**: **ユーザーに確認を促す**こと。責任の割り当て・是正・運用ルールの見直しなど、**人の判断**に回す。**遷移の事実そのものはインプリントに残り**、事後の監査（③）とも連携する。
- **事前拒否との違い**: **「想定アクターのみ実行可」で遷移を弾く**方式ではなく、**想定外なら知らせる**（**ブロック第一ではない**。**遷移アラート ≒ 事前拒否 ではない**）。

### 「権限」という語について

- **許可（承認）の儀式**ではなく、**許可される矢印（状態機械）**・**想定外時の遷移アラート**・**事後のインプリント** に言葉を分けると、Issue 管理の本質に寄せやすい。

## 稟議・承認との違い（概念）

| 観点 | 稟議・承認（本プロジェクトでは対象外にシフト） | Issue のステータス遷移（本設計） |
|------|--------------------------------|---------------------------|
| 問い | 誰が**許可したか**（責任の所在・証跡） | **経路（形）**・**想定外 actor への遷移アラート**・**事後監査** |
| 典型 | 承認ログ | カンバン上の経路制約に加え、**想定外の完了遷移**で **リーダー等へ遷移アラート** |

本リポジトリに存在する **承認ステップ系（workflow_steps/approval_objects/issue_approvals）** は、上記方針のもとではレガシー扱いとして **削除対象** とする。

## ユースケース例（遷移の順序・アラート）

- **チームリーダー**が Issue を「確認済」へ、続けて **課長** が「完了（Close）」へ、といった**複数ステータスの順序**は、承認ログではなく **遷移ルール（形）**として表す。
- **完了**へ遷移した actor が「課長グループ」に含まれない場合に **遷移アラート** を飛ばし、**PM グループ**などへメールする、といった **想定アクターと実態のずれ**の検知に使う。

## 設計の軸（議論リスト）

| 軸 | 論点 |
|----|------|
| **単位** | 組織全体の「階層レベル」だけで足りるか。グループごとに「課長相当」が違うなら **グループとセット**が必要。 |
| **階層 vs 多所属** | 数値レベルで「主任以上」を一発で書けることと、「同一人物が複数グループに所属」だけの単純さのトレードオフ。併用もあり得る。 |
| **グループ** | 「営業部の課長のみ」を満たすには、**Issue またはユーザー側にグループ文脈**が必要になる場合がある。 |
| **ルールの載せ先** | **遷移先ステータスごと** / **遷移（from→to）ごと** など、表現力と複雑さのトレードオフ。 |

## 候補モデル（比較。個人 ID のみへの固定は避ける）

以下は **「想定アクター」や遷移アラート条件**の表現に使う候補であり、**遷移を拒否する**ためのルール定義ではない（ゲートキーパー型の事前拒否は採用しない）。

### 案 A — ステータス + 組織内レベル

- **想定アクター**の例: 組織内で `level >= N` の者のみ「想定」とみなし、それ以外が遷移したらアラート、等。
- **限界**: 「営業部の課長のみ」は、**組織全体の1本のレベル列**だけでは表しにくい。

### 案 B — 権限グループ（命名例）+ ユーザー多対多

- 「営業部・管理職」など**グループを定義**し、ユーザをメンバーに。**想定**: 「このグループのメンバーである actor の完了遷移はアラート不要」等。
- **利点**: 異動はメンバー更新で済み、**個人 ID に Close を結ばない**。

### 案 C — グループ × レベル（またはグループスコープ付きグループ）

- `group_id` と rank／レベルを組み合わせて **想定アクター**や **遷移アラート条件**を定義。
- **要件「営業部の課長のみを想定」**に直結しやすい。

### 案 D — ハイブリッド

- 例: シンプルな遷移はレベルのみ、重要な遷移だけグループスコープを付与、等。
