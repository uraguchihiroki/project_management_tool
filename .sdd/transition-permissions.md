# ステータス遷移の権限（設計メモ・候補比較）

Issue 管理を主目的とするにあたり、**稟議・承認ワークフロー（誰が許可したかの証跡）とは別**に、「**誰がステータスを変更してよいか**」を制御する必要がある。

- **目的の例**: チケットを勝手に Close されないこと（特にプロジェクトマネジメント上の問題）。**Close を特定ユーザー ID にだけ結び付けない**こと（人事異動で権限が個人に張り付くと運用が破綻しやすい）。
- **要件の例**: 「**営業部の課長**だけが Close できる」のように、**部署 × 役割（名称は未確定）**で表現できること。

用語は **「役職」に固定しない**。階層付きグループ・名前付きグループ・部署スコープなど、下記を議論のうえ確定する（**第1案はまだ決めない / 1-D**）。

---

## 承認フローとの違い（概念）

| 観点 | 稟議・承認（本プロジェクトでは対象外にシフト） | Issue のステータス遷移権限 |
|------|--------------------------------|---------------------------|
| 問い | 誰が**許可したか**（責任の所在・証跡） | **誰がやるか**／誰がボード上の列を変えてよいか |
| 典型 | 承認ステップ、承認ログ | カンバン上の「確認済 → 完了」などの**門番** |

本リポジトリに存在する **Workflow / IssueApproval 等の設計・実装は、上記方針のもとではレガシー扱いとし、ドキュメントからは切り離して記述する**（コード削除は別タスク）。

---

## ユースケース例（門番・順序）

- **チームリーダー**が Issue を「確認済」へ、続けて **課長** が「完了（Close）」へ、といった**複数ステータスの順序**は、承認ログではなく **遷移ルール**として表す。

---

## 設計の軸（議論リスト）

| 軸 | 論点 |
|----|------|
| **単位** | 組織全体の「階層レベル」だけで足りるか。部署ごとに「課長相当」が違うなら **部署とセット**が必要。 |
| **階層 vs 多所属** | 数値レベルで「主任以上」を一発で書けることと、「同一人物が複数グループに所属」だけの単純さのトレードオフ。併用もあり得る。 |
| **部署** | 「営業部の課長のみ」を満たすには、**Issue またはユーザー側に部署文脈**が必要になる場合がある。 |
| **ルールの載せ先** | **遷移先ステータスごと** / **遷移（from→to）ごと** など、表現力と複雑さのトレードオフ。 |

---

## 候補モデル（比較。個人 ID のみへの固定は避ける）

### 案 A — ステータス + 組織内レベル

- 各ステータスへ遷移できるのは、組織内で `level >= N` の者のみ、など。
- **限界**: 「営業部の課長のみ」は、**組織全体の1本のレベル列**だけでは表しにくい。

### 案 B — 権限グループ（命名例）+ ユーザー多対多

- 「営業部・管理職」など**グループを定義**し、ユーザをメンバーに。遷移ルールは「このグループのメンバーだけ」。
- **利点**: 異動はメンバー更新で済み、**個人 ID に Close を結ばない**。

### 案 C — 部署 × レベル（または部署スコープ付きグループ）

- `department_id` と rank／レベルを組み合わせて門番を定義。
- **要件「営業部の課長のみ Close」**に直結しやすい。

### 案 D — ハイブリッド

- 例: シンプルな遷移はレベルのみ、重要な遷移だけ部署スコープを付与、等。

---

## 次に決めたいこと（チェックリスト）

1. Close / 重要ステータスのルールを、**部署スコープ必須**にするか、**組織単位のレベルだけ**も許すか。
2. **主たる単位**を「階層レベル」「名前付きグループ」「部署×役級」のどれに置くか（併用なら境界）。
3. **Issue が属する「部署文脈」**をどう取るか（Issue に department、プロジェクトに既定部署、担当者の部署、等）。
